# クラウド実験サイト - CRUD & 認証設計 学習プロジェクト

職業訓練校でPython/Javaの基礎文法を学びながら、実際に手を動かして「動くもの」を作るために個人で開発した実験用プロジェクトです。
CRUD操作の実装から、フロントエンド/バックエンドの分離、認証設計、テーブル設計、テスト、クラウドデプロイまでを一通り自作しました。

## 🔗 デモ

- フロントエンド(Vercel): https://store-iphone-portfolio.vercel.app/index.html
- バックエンドAPI(Render): https://store-iphone.onrender.com

ウェブサイト閲覧は誰でも可能です。DB保存・編集・更新・削除には管理者ログインが必要です。

> ⚠️ バックエンド(Render)は無料プランで運用しているため、しばらくアクセスが無いとサーバーがスリープします。
> 初回アクセス時、応答までに20〜30秒ほどかかることがありますが、これはRenderの無料プランの仕様によるものです。

## 構成図

```
┌─────────────┐          ┌────────────────────┐           ┌──────────────┐
│   Vercel    │ ------>  │  Render (Flask)    │ ------>   │  Supabase    │
│  (フロント)  │  fetch   │  認証・API プロキシ │  service  │ (PostgreSQL) │
│  HTML/JS    │ <------  │                    │  _role    │  RLS で保護   │
└─────────────┘          └────────────────────┘           └──────────────┘
```

- **フロントエンド(Vercel)**: 静的なHTML/JS。GitHub連携で自動デプロイ、ブランチごとにプレビュー環境あり
- **バックエンド(Render)**: Flaskで自作したAPIサーバー。認証とSupabaseへの通信を仲介
- **DB(Supabase)**: PostgreSQL。RLS(Row Level Security)を有効化し、anonキーからの直接アクセスを完全に遮断。テーブル間は外部キーで関連付け

## なぜこの構成にしたか(設計意図)

最初はフロントから直接Supabaseに接続するだけの構成でしたが、それだと「anonキーさえあれば誰でもDBを操作できる」状態になってしまうことに気づき、以下の段階を踏んで設計を見直しました。

1. **RLSを有効化**し、ポリシーを設定しないことでanonキーからの読み書きを完全ブロック
2. **Flaskをプロキシサーバーとして自作**し、Supabaseの`service_role_key`(RLSを無視できる強い権限の鍵)をサーバー側の環境変数にのみ保持
3. フロントのJSに秘密情報を一切残さないため、**パスコード認証をCookieではなくトークン方式**で実装
   - 理由: Cookieベースのクロスサイト認証は、SafariのITP(Intelligent Tracking Prevention)によりブロックされることが判明したため
4. 当初は「ログイン画面へページ遷移→ログイン後に元のページへリダイレクト」という方式にしていたが、**別ドメイン(Render)への画面遷移がPWAのscopeを外れてしまい、standalone表示が崩れる**問題が発生
   - 対策として、画面遷移せずに`fetch`でログインAPIを直接呼び出す方式に変更。未認証の書き込み操作を検知すると、その場でパスコード入力ポップアップ(自作モーダル)を表示し、認証後は同じ画面のまま処理を継続する
   - 発行されたトークンは`localStorage`に保存し、以降のリクエストで`Authorization: Bearer`ヘッダーとして送信
5. `alert` / `confirm` / `prompt`はブラウザの「今後表示しない」設定でブロックされることがあるため、**自作のモーダルコンポーネント**に置き換え、ブラウザ設定に左右されない安定した挙動にした
6. 認証トークンは当初`itsdangerous`(Flask独自の署名付きトークン)で自作していたが、業界標準規格である**JWT(PyJWT)に置き換え**、ヘッダー・ペイロード・署名という標準構造を採用
7. `pytest`によるテスト導入の過程で、**秘密鍵の強度不足**(HMAC鍵が推奨32バイト未満)と**ローカル/本番環境変数の同期漏れ**という、実運用に関わる問題を発見・修正

## 実装した機能

- [x] CRUD全操作(Create / Read / Update / Delete)
- [x] SQLite版バックエンド(ローカル学習用、Flask + SQLite)
- [x] Supabase版バックエンド(クラウドDB、PostgreSQL)
- [x] RLSによるDBアクセス制御
- [x] Flaskによる認証プロキシ(パスコード認証 + JWT)
- [x] ログイン試行回数の制限(総当たり攻撃対策、IPごとに5回失敗で5分ロック)
- [x] タイミング攻撃対策(`secrets.compare_digest`による安全な文字列比較)
- [x] alert/confirm/promptを自作モーダルに置き換え(ブラウザ設定によるブロックを回避)
- [x] XSS対策(`textContent`を使用し、`innerHTML`による直接埋め込みを回避)
- [x] テーブル関連付け(外部キー + JOIN、投稿者テーブルとの紐付け)
- [x] ページネーション(offset/limitによる「もっと見る」形式の追加読み込み)
- [x] 作成日時・更新日時の自動記録(DBトリガーによる`updated_at`自動更新)
- [x] pytestによる自動テスト(認証必須のエンドポイント、バリデーションの検証)
- [x] Gitブランチ運用 + Vercelプレビューデプロイによる、本番を汚さない検証フロー
- [x] npm scriptsによるローカル/本番環境設定のワンコマンド切り替え
- [x] Dockerによるコンテナ化(ローカルでのビルド・実行を検証)
- [x] Renderへの本番デプロイ(24時間稼働)

## 今後の課題(自覚している改善点)

- パスコードは4桁の簡易認証であり、本番運用であればユーザー登録・パスワードハッシュ化・多要素認証などが必要
- ログイン試行回数の制限はサーバーのメモリ上に保持しているため、Renderの再起動でリセットされる(本番運用ならRedis等の永続ストレージが必要)
- Renderの無料プランはアクセスが無いとスリープするため、初回アクセス時に応答まで時間がかかることがある
- テストは主要な認証・バリデーションのみカバーしており、全エンドポイントを網羅できていない
- Dockerはローカルでの動作確認まで実施し、Render既存サービスのランタイム変更に制約(ダッシュボードからの直接切替不可)があることを確認した段階で、本番導入は見送り
- CI/CD(GitHub Actions等によるテスト・デプロイの完全自動化)は未導入

## 学習の経緯

このプロジェクトは、訓練校の授業(Python/Javaの文法学習)と並行して、「文法を知っているだけでは動くものが作れない」という課題感から始めました。
Supabaseとの接続・RLS設定・Flaskでの自作API・認証設計・クラウドデプロイと、つまずきながら一つずつ原因を切り分けて解決してきた過程そのものが、このプロジェクトの一番の成果だと考えています。

その後もテーブル設計(外部キー・JOIN)、ページネーション、Gitブランチ運用、Docker、そしてテストコードの導入と、機能追加のたびに新しい概念を実際に手で動かして確かめてきました。
特に、認証方式を3回作り直した経験(Cookie→署名付きトークン→JWT)や、テストを書く過程で秘密鍵の強度不足という実際のセキュリティ課題を発見できたことは、単に動くものを作るだけでなく「なぜそう設計すべきか」を考える視点に繋がったと考えています。
